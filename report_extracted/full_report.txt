Chapter 4
DESIGN AND ARCHITECTURE
4. Design and Architecture
This chapter outlines the full design and architecture specification of the Event Sphere system as outlined in the Software Design Description (SDD). It outlines the three-layer system architecture, data model representation and entity relationships, workflow model represented in the activity diagram, and formal design models such as sequence diagram, class diagram and data flow diagrams. Collectively, these components offer a holistic view of the structure and its behaviour that underpins the design of the system.
4.1 System Architecture
Event Sphere is based on the three-layer client-server architecture with cloud services. This ensures real-time event notifications, secure delivery of data, a scalable backend service and seamless communication between all user types. The three layers of the Event Sphere architecture are the Presentation, Application and Data Layer, with clearly demarcated responsibilities and boundaries.
4.1.1 Presentation Layer
The Presentation Layer is the mobile application interface of Event Sphere and constitutes all of the user interface elements that the end user interacts with on their Android phone. It is implemented entirely using Flutter and Dart, a framework for building user interfaces entirely from widgets that are rendered with high fidelity and responsiveness.
The Presentation Layer is structured into three different role-based interfaces, rendered at runtime depending on the role of the logged-in user:
Student Interface
The Student interface allows access to the event discovery and browsing screen, the event registration module with one-tap registration, the QR code scanner for attendance logging on the day of the event, the bookmarks list for storing events of interest, the notifications module to view event registrations and reminders, and the user profile screen to update user details and view participation history.
Faculty Interface
The Faculty interface offers the event creation and management form, the pending events list indicating the current status of submissions, the participant list and attendance screen for approved events, the expense management module with receipt upload feature, and the analytics dashboard displaying participation trends and attendance data for events created by Faculty members.
Admin Interface
The Admin interface offers access to the user management panel for adding, editing and deleting Faculty and Student users, the event approval queue for reviewing and approving pending events, the system-wide analytics and participation reports dashboard, the expense approval process, the content moderation tools for deleting invalid or expired events, and the system administration screen for configuration and activity log.
All user interface screens in the Presentation Layer include form validation prior to transmitting data to the backend server, real-time updates triggered by Firebase Firestore database listeners, and a consistent user experience with common themes such as consistent use of color, navigation, and error messages across all modules.
4.1.2 Application Layer
The Application Layer is the business and service integration layer of Event Sphere. This layer is located between the Presentation Layer and the Data Layer, and is responsible for handling user interactions, applying business rules, orchestrating service operations and managing the application state. The Application Layer is implemented in the Flutter application as service and state management classes.
Authentication Service
The Authentication Service handles all user authentication via Firebase Authentication. This includes user registration, password verification, session tokens, password resets and email verification. Most importantly, it is responsible for Role-Based Access Control (RBAC) via reading the user role from Firestore after login and providing the appropriate role-based view to the application. Only authenticated users with a role can go beyond the login screen.
Event Service
The Event Service handles the entire event lifecycle. It manages event creation (including input validation and Firestore write) by Faculty, event retrieval (filtered by status=Approved) for student browsing, event status update (approval/rejection with comments) by Admin, and real-time event data synchronization to all devices via Firestore listeners. Furthermore, the Event Service integrates with the Notification Service to send automatic push notifications at various events lifecycle stages.
Registration and Attendance Service
The Registration Service handles student registration for events using Firestore transactions to check event seat availability and reduce it in an atomic way, avoiding overbookings even if multiple clients try to register at the same time. It records student sign-ups, ensures that students can only sign up for one event and sends confirmation emails. The Attendance Service generates QR codes for each registered event and validates QR code scans, ensuring the student is registered, the event matches, and that the student is not checked in twice, then logs the attendance in Firestore.
Notification Service
The Notification Service uses Firebase Cloud Messaging (FCM) to provide instant notifications to all users. It is automatically triggered by other services on some important events such as event creation, approval, rejection, student enrolment, 24-hour event reminder, schedule change and system announcement. Notifications are also saved in the Firestore Notifications collection to be displayed in the user's notification center in the app.
Expense Service
The Expense Service handles event expenses. Faculty create expense records, including the value, category, description, and upload the receipt photo to Firebase Storage. The service records expense information in Firestore and sends a notification for Admin approval. The service receives the Admin's approval or rejection and updates the expense status in Firestore and sends the Faculty member a notification of the decision.
4.1.3 Data Layer
The Data Layer is the component of Event Sphere that handles all persistent data storage, access and management. It consists of three different back-end services that offer real-time synchronisation, relational data storage and management, media storage and secure access control.
NoSQL Database (Firebase Firestore)
The main real-time database of Event Sphere is Firebase Cloud Firestore. It maintains the data of all the major components of the application in a NoSQL document-collection format in the following collections: Users (storing Admin, Faculty, and Student information), Events (storing all events with status, organizer, and approval details), Registrations (storing student sign-up details), Attendance (storing QR code-verified student attendance records), Notifications (storing system-generated notifications), Expenses (storing faculty expense claims), and Societies (storing university societies information). Firestore's real-time listeners automatically update the database changes in real-time to all connected clients without needing to refresh.
Supabase (Supplementary Backend)
Supabase is used as an extended backend system to address additional relational data storage needs that extend beyond Firestore's NoSQL capabilities. It offers relational storage and can be used as a second storage layer to support data operations that require a relational schema (such as complex queries and joins between multiple entities).
Firebase Storage (Media Storage)
Firebase Storage is used for storing all binary files in Event Sphere. It handles the storage of event poster images uploaded by Faculty, the storage of images and PDFs for expense receipts, certificates of attendance for students who have registered for events, and any other media used for promotion. All files are stored with secure tokens and their download URLs are stored in the Firestore document records for the application to access.
Firebase Security Rules
Firebase Security Rules are set for Firestore and Firebase Storage to implement role-based data access controls. These determine that Student users can only read approved events and their own event registration and attendance records, Faculty users can only write events they created, and that only Admin users can access all collections with full read and write rights. Data communication between the mobile client and Firebase services is secured with SSL/TLS.

The data structure is defined using the data dictionary, entity relationships and the conceptual Entity-Relationship (ER) model.

4.2.3 ER Diagram Overview
The conceptual Entity-Relationship (ER) model of Event Sphere has the Event entity as the central source of data for other entities. The most frequently referenced entity is the User entity, as a foreign key of Events (as event organizer), Registrations (as event participant), Attendance (as event attendee), Notifications (as event recipient) and Expense (as expense submitter). This prominence of User is an outcome of the system's role-based architecture where every major action is initiated by and can be traced back to a user.
Event connects to Registration, Attendance, Notification, and Expense, representing the entire event lifecycle: a Faculty User creates an Event, Students register using a Registration record, check in using an Attendance record, Notification records are automatically created at each step in the lifecycle, and Faculty submit Expense records for the event. The Society entity links to Event through a one-to-many relationship, reflecting that each university society can host numerous events each year.
The ER model is implemented in Firebase Firestore as a document-collection database where each entity is a collection, a relationship to another entity is represented as a reference to a document in that entity, and Firestore Security Rules control access to collections and documents based on a user's role.
4.3 Process Flow / Representation
This section outlines the process flow of Event Sphere in the Activity Diagram, which represents the whole system process flow from user login to post-event processing. The process flow illustrates the flow of data and control in response to user and system events.
4.3.1 Activity Diagram
The Activity Diagram (Figure 1 in the SDD) is a global overview of the workflow associated with the Event Sphere application, capturing the interactions of the three users' roles from system entry to after the event. The diagram is divided into five stages: Login/Authentication, Admin/Faculty Event Management, Student Participation, Event Day and Post-Event.
The diagram starts with the end-user launching the mobile app and authenticating their credentials. The user's credentials are authenticated by Firebase Authentication and an RBAC check is performed to verify the user's role. Depending on the role, one of the dashboards (Admin, Faculty, or Student) is displayed. The dashboards have access to the role-specific modules in the Presentation Layer.
For the Admin/Faculty roles, the user manages events. The Faculty creates an event, which is set to Pending status on Firestore and goes into the Admin approval flow. The Admin approves/rejects the event. If approved, the event is automatically published to the list of events the students can see in real-time through Firestore. A rejected event triggers a notification to the Faculty organizer, with the reason for rejection.
In the Student path, the student scans the list of approved events, searches or filters, clicks on an event of choice and registers with a single click. The system checks for available places, logs the registration and notifies the student. The student can also "bookmark" the event or add it to their device calendar.
4.3.2 General Workflow
The Event Sphere workflow as a whole is as follows:
User Login: The user launches the app, logs in, and is authenticated by Firebase Authentication. RBAC checks and displays the user's dashboard.
Event Creation: A Faculty member proposes an event by entering the necessary information. The system checks for errors and saves the event into Firestore with the status set to Pending, and sends the Admin a notification.
Event Approval: The Admin views a list of events pending approval, approves/rejects the event and the system updates the status of the event and notifies the Faculty member of the event.
Event Discovery: The student views the listing of the approved events, searches and filters events based on the search criteria, sees the full event details on the event card containing the event title, description, location, date and the organizer of the event.
Student Registration: The student taps to register for an event. The event capacity is checked atomically, the registration is persisted, the remaining capacity is reduced and a confirmation is sent.
QR Attendance: On the day of the event, a QR code specific to the event is created. Students use the QR code to check in. It verifies the scan with the registration data, prohibits multiple check-ins and logs the attendance in Firestore.
After the Event: After the event, the system generates participation certificates for those eligible, asks for student feedback and ratings, stores all feedback in Firestore, and sets the event status to Completed.
Reports and Analytics: Admin & Faculty can view the reports dashboard to check data on participation, attendance, department wise participation, and export events in the form of reports.
4.4 Design Models
Design models of Event Sphere represent the system's structure and behavior. They provide visual architectural representations that are used as a guide to the development team and ensure system modularity, scalability and maintainability. The next subsubsections discuss the Sequence Diagram and the Class Diagram, which represent the dynamic and static aspects of the system, respectively.
4.4.1 Sequence Diagram
The Sequence Diagram (Figure 2 in the SDD) illustrates how the components interact during the Event Approval Workflow process, one of the main processes in Event Sphere. This sequence diagram has four components: Faculty actor, UI Layer (Flutter front-end), EventService (application layer), Firestore events collection (data layer) and NotificationService.
The sequence diagram begins with a Faculty member opening the Pending Events screen. The UI Layer makes a request to the EventService to get pending events (getPendingEvents()) which returns all events that are "pending" from Firestore. Firestore returns the pending events which are then displayed by the UI to the Faculty member.
If the Faculty member wants to view more details about an event, the UI calls the getEventDetails(eventId) method of the EventService. The EventService queries Firestore to get the full event document containing the event title, venue, date, organizer, description and notes and returns these to the UI for display.
When the Faculty member has reviewed the event, they click the Approve button. The UI calls the EventService's approveEvent(eventId, approvalNotes) method. If the current event status is neither "approved" nor "rejected", then the EventService atomically updates the Firestore event document, setting the status to "approved", and with the approval date and approval notes. Firestore confirms the update.
Once the event status has been successfully updated, the EventService sends an event to the NotificationService to send a send ApprovalNotification(organizerId, eventId, approved=true) message. The NotificationService creates a new document in the Firestore Notifications collection for the event organizer.Once the notification is created, the NotificationService returns a success message to the EventService, which then returns an "approval success" message to the UI Layer. The Front End layer prompts a message "Event Approved Successfully" for the Faculty member and completes the sequence.
This interaction illustrates the collaborative, real-time and secure event approval process that involves all three architectural layers of Event Sphere, and guarantees data integrity and real-time notifications to stakeholders.
4.4.2 Class Diagram
The Class Diagram (Figure 3 in the SDD) is a static description of the Event Sphere system, which defines all the major entities of the system as object-oriented classes and their relationships. The class diagram follows the UML class diagram and shows the object-oriented structure of Event Sphere.
Core Classes
The class diagram has the User class as the parent class. It stores common attributes of a user: uid, email, name, role, department, createdAt and updatedAt. It has methods such as login(), logout(), updateProfile() and getPermissions(). The User class is a superclass for three different types of users: Admin, Faculty and Student, which inherit methods from the User class. This defines manageUsers(), manageSocieties() and viewAllEvents(). Faculty defines createEvent(), approveEvent() and manageExpenses(). Student defines registerEvent(), viewEvents() and checkIn().
Event Class
Event class defines all the attributes of a university event: eventId, title, description, venue, date, capacity, organizerId, societyId, status, approvalNotes, approvalDate, and posterUrl. It has create(), update(), delete(), approve(), generateQR(), and getAttendance() methods. Event class is linked to Registration (one-to-many), Attendance (one-to-many), Notification (one-to-many) and Society (many-to-one) classes.
Registration Class
The Registration class represents a student's registration for an event, and is related to the User and Event classes. This class has fields registrationId, eventId, userId, registrationDate and status. It has methods register(), cancel(), checkIn() and getStatus(). A Registration record is a student's event registration record and status can be Registered or Cancelled.
Attendance Class
The Attendance class is used to capture student attendance at events. It has properties like attendanceId, eventId, userId, checkInTime, verified (Boolean) and certificateEligible (Boolean). Methods of the class are scanQR(), markAttendance() and issueCertificate(). When the QR scan is verified, the verified is set to true and the certificateEligible is set according to the attendance status.
Notification Class
The Notification class is a user notification. It has the following properties: notificationId, userId, message, type (Approval, Reminder, Update or System), date and seen (Boolean). It has send(), markAsRead() and delete() methods. The Notification is created by the NotificationService from system events such as event approvals, event registrations and reminders.
Expense Class
The Expense class is used in the event financial management process. It holds expenseId, eventId, amount, category, receiptUrl, status (Pending, Approved, Rejected) and the submittedBy reference. Its operations are create(), approve(), reject() and uploadReceipt(). Expenses are submitted by Faculty and approved by Admin in the expense management process.
Society Class
The Society class models university student societies hosting events. It holds societyId, name, description, presidentId and memberCount. It has methods create(), addMember(), removeMember(), and updateDetails(). A Society can have many Events associated with it, as per the Event-to-Society relationship.
The Class Diagram collectively shows how these seven classes work together to support the entire scope of Event Sphere's event management lifecycle, and offer an object-oriented structural overview that guided the development of Event Sphere.
4.4.3 Data Flow Diagrams
DFD Level 0 - Context Diagram
The DFD Level 0 diagram (Figure 4 in the SDD) is a top-level diagram that shows Event Sphere as a single process with external interactions to entities and data stores in the cloud. The three user roles (Admin, Faculty, Student) supply information to the system (event data input, registration requests, and administrative operations) and receive information from the system in the form of screens, notifications, certificates, and reports. Event Sphere system also interacts with three data stores: Firestore Database for reading and writing all structured data related to the application; Firebase Storage for storing and retrieving event posters, certificates, receipts and other documents; and Google Gemini AI for chat-based queries and AI-generated responses to provide the built-in AI assistance feature.
DFD Level 1 - Event Management
The Level 1 DFD (Figure 5 in the SDD) divides the high-level event management process into five sub-processes: Create Event (1.0), Store Event to DB (2.0), Send Approval Notification (3.0), Update Event Status (4.0) and Publish or Notify (5.0). An Admin or Faculty user enters event information via subprocess 1.0, which validates the event details and passes this information to subprocess 2.0, where a new event record is added to the Firestore Events collection with a status of PENDING. Subprocess 3.0 automatically notifies the approving faculty member. When reviewed, the the approval or rejection moves to subprocess 4.0 which updates the event status in Firestore to either APPROVED or REJECTED, along with timestamps and notes. For rejected events, a notification is sent to the organizer via the same subprocess.

IMPLEMENTATION
5. Implementation
The implementation phase is responsible for turning the design architecture and requirements of Event Sphere into a fully functional and multi-role mobile application. The implementation is an Android mobile application using Flutter and Dart and a cloud-based backend using Firebase Authentication, Cloud Firestore, Firebase Cloud Messaging, Firebase Storage and Supabase. The system includes three different roles (Admin, Faculty, and Student) with a dedicated screen and features.
The system has been built in an Agile manner with the following components for user authentication and registration, event creation, approval by admin, student registration, attendance via QR code, real-time notifications, expense management, and reporting. The system implementation adheres to the modular architecture presented in Chapter 4, with real-time data integration, role-based navigation and optimal and consistent mobile user interface design across all Android devices.

If a match is found, an error message is returned and no attendance record is saved.

The advent of mobile and cloud computing technologies has revolutionised information management and dissemination in organisations. Universities are vibrant campuses, hosting a variety of events throughout the year. There is no digital, integrated system for event management for university events that links event creation, administrative approval, student event registration and attendance, expenses and real-time communication and provides access to all this in a role-based, integrated system. The application must have Role-based access control (RBAC) that must implement, at both client and server side, strict access control for Admin, Faculty and Student roles.  It must leverage Google Maps API for event location, mapping and venue management.
The QR code system must allow automatic generation of QR codes for each event and seamlessly include a QR code scanner to check attendance.
Functional Requirements
The system shall provide full user lifecycle management in terms of registration, login, user profile update and user role management for all user roles. 
Academics must be able to add, edit and submit events which are reviewed and approved by administrators. 
Event podcasts must be approved or rejected by admins with notes sent to the event organisers via push notifications. 
Students have access to browse, search and filter, register for and bookmark events through the mobile app. 
The expenses module should provide an option for faculty to upload expenses (along with receipts) and for administrators to approve/reject expenses.
Admin dashboard must include event analytics, user management and event reports download.
Non-Functional Requirements
Performance: Event list loading time should be under two seconds, with a user load of at least 500 users. 
 Scalability: The server architecture should allow for horizontal scaling for a growing number of users, events, and records. 
 Security: Data collected in the system should be transmitted using SSL/TLS, and Firebase Security Rules should be used to enforce role-based database access control.
  Usability: The user interface should be simple and user-friendly, not requiring technical expertise for any role.
  Reliability: The system should have at least 99.9% uptime during term time.
  Maintainability: The system must be structured according to OOP principles and must have well separated modules for ease of modification.
2.3 Current System
Providing a background on the current system is needed to better understand the level of improvement brought by Event Sphere. The current system for university events is a mixture of manual processes, ad hoc communications and digital systems.
2.3.1 How the Current System Operates
The current event management process is an informal one which differs across departments and individuals:
Event Announcement: Faculty circulate announcements via WhatsApp groups or Facebook pages and/or physical notice boards and email. Content is not in a consistent format or kept in a central database. 
Registrations: Links to Google Forms or handwritten registration sheets are provided to students. Neither offers confirmation or checks for duplicate enrollments.
Approval: Event approval is done informally via phone or email without any formal approval process or audit trail. 
Check In: Students are checked in on paper lists, which is time-consuming, inefficient, and resource-wasteful. 
event Budget: There is no formal system for the submission or the approval of event expenses, resulting in accountability issues.
After the Event: Data is kept ad-hoc in spreadsheets or individual files with no institutional level data retention or analytics.
2.3.2 Limitations of the Current System
No Consolidation: Events are not centralised - information is spread across multiple media platforms.
Weaker Communication: Knowledge dissemination through WhatsApp and social media channels has limitations due to group restrictions and information overload.
Lack of Approval Process: Events are not reviewed and approved before publishing, so it can't be guaranteed they meet institution standards.
Paper-based (and prone to errors) Event Registration: Registration is manual and cannot adhere to capacity limits, check for duplicate entries, or automatically confirm attendance. 
Inefficient Attendance Monitoring: Manual attendance tracking is time consuming, inaccurate and requires staff support. 
No Expense Tracking: Expenses are monitored informally and no process is followed for expense submission and approval. 
Lack of Analytics: There is no system to record and analyse event attendance data for planning and evaluation. 
Fragmented and Lost Data: The fragmented nature of the files being stored and the loss of data means it is often unavailable for reporting.

3.1.1 Actors
There are four key actors in Event Sphere, with different roles and interaction with the system:
Admin
Event Sphere's administrator has the highest level of authority. The Admin is in charge of user management, reviewing and approving or denying event submissions from Faculty, maintenance and configuration of the system, moderating the content of the system, and reviewing all reports, including the statistical reports of attendance. The Admin is involved with all five main use cases.
Faculty
Faculty are event organisers. They can propose and submit events, manage events they have organized, view participant registrations for events they have proposed and organized, submit expenses for events they have organized and access reports and analytics for their events. The Event Management, Event Participation, Reports and Analytics, and Expense Management use cases are the main interactions faculty have with the system.
Student
The target audience of Event Sphere is students. They attend and find approved events, sign up to attend events with one click, get push notifications for event changes and reminders, mark events they will attend, and confirm their attendance on the day of the event using a QR code. Students engage with the Event Participation and Notifications use cases.
System
The System actor is the automated system processes that run in the background. The system is responsible for sending Firebase Cloud Messaging (FCM) notifications, creating unique QR codes for events that are approved, checking whether a participant is attending an event, processing status requests for events, backing up data, and maintaining a log of all system activities. It facilitates and supports the interactions of all other actors in the event process.
3.1.2 Primary Use Cases
The Event Sphere system is made up of five primary use cases, each corresponding to a key functional component of the application:
UC-1 User Management: Represents all of the user account lifecycle functions (create, update, delete, assign role) by the Admin with the support of System.
UC-2 Event Management: Manages the entire event life cycle from Faculty creation and submission to approval by the Admin and publication to students, involving Admin, Faculty and System actors.
UC-3 Event Participation: Involves student event registration, digital one-tap sign-ups, QR code based attendance check and certificate eligibility management by Student, Faculty and System actors. 
UC-4 Reports and Analytics: Covers creation, viewing and exporting of event participation, attendance, and engagement reports, and analytics, involving Admin and Faculty, and System.
UC-5 System Administration and Maintenance: Covers system configuration settings, data backup, log activity creation, system issue resolution and system "well-being" management by Admin, with System support.
3.1.3 Use Case Relationships
Event Management use case includes optional Bookmark and Event Sharing.
Event Participation has an optional QR Code component for check-in.
Event Participation includes the Notifications module for confirmations, reminders and notifications.
The Data Management module has dependency with the Event Management, Event Participation and Reports and Analytics.
System Administration and Maintenance aggregates with the Log and Track System Activities module.

CONCLUSION AND FUTURE WORK
7. Conclusion and Future Work
7.1 Conclusion
Event Sphere is a successful example of how the latest mobile technologies, cloud-based back-end services and real-time data synchronization can be integrated to build a highly accessible, efficient, and role-based university event management system. The app solves a well documented problem at COMSATS University Islamabad, Sahiwal Campus - the lack of a centralised, online system for managing the entire event life cycle from proposal creation by faculty and approval by administrators, to student registration, attendance tracking using QR codes, and analytics after the event.
The development of this system was based on the Agile Software Development Lifecycle and was done in nine well-defined phases, each focusing on a particular module of the application. This allowed the development team to develop, test and refine each module in isolation but ensure they were integrated and tested at regular intervals across the whole system. The resulting system is a complete Android mobile app that caters for three types of users:
Admin: Manages users, approves or rejects events, reviews expenses, configures the system and assesses institutional analytics.
Faculty: Event management, monitoring of participants, expense submissions and analytics for events they organize.
Student: Event browsing, single-click event registration, QR attendance confirmation, push notification, and event favourite.
The system satisfies all the functional and non-functional requirements in the SRS, including:
Responsive system: Event registration acknowledged in 1.4 seconds on average; QR scanning for attendance acknowledged in 2.1 seconds on average.
Event synchronization: Firebase Firestore listeners broadcast changes to the state of events and other data updates to all devices within 500 milliseconds.
Scalable backend: Firebase and Supabase cloud infrastructure handles up to 500 users simultaneously with consistent performance, achieving the desired scalability.
Secure permissions: Firebase Authentication and Firebase Security Rules manage rigorous role-based access for the three user roles for the app and the database.
Automated notifications: Firebase Cloud Messaging system pushes notifications for all specified events within 2.8 seconds on average.
QR-based attendance: The multi-validated QR attendance module eliminates proxy attendance and multiple check-ins with 100% efficiency in all use cases.
Modular and maintainable code: Flutter and Dart code is written using object-oriented design principles, and separated into service and user interface (UI) modules for ease of maintenance.
In conclusion, Event Sphere meets all the objectives outlined in the SDD and represents a solid and scalable foundation to build a university event management ecosystem that can be further extended, improved and implemented in multiple campuses.
7.2 Limitations
While Event Sphere is stable across the broad range of scenarios tested, during the development and testing process, we identified a number of limitations that can be improved in the future:
7.2.1 Internet Connectivity Dependency
Event Sphere is currently an online application, requiring internet connectivity for almost all of its features, including searching and viewing events, registering for events, using QR codes to track attendance, sending notifications, as well as administrative tasks. The application is not usable in low- or no-signal areas. Although Firebase Firestore provides some offline capabilities, these were not fully set up in this version, so event information may not be available offline. This makes the app less useful in the context of university where network connectivity may be unreliable.
7.2.2 Android Platform Only
Event Sphere is currently only compatible with Android devices on Android 8.0+. The first release doesn't support iOS devices because of the resources required for platform-specific development and testing. This restricts the use of the app to students and teachers with iPhone devices. Although Flutter's cross-platform ability means that the iOS version can be developed without rewriting the code, testing, Firebase configuration and App Store deployment issues must be resolved to release the iOS version.
7.2.3 QR Scanner Performance Under Low Lighting
During testing, there was a slight performance degradation of the QR code scanner module in very low lighting conditions. When the event space was dimly lit, some students needed to scan their QR code multiple times for the code to be captured by the camera. The mobile scanner package works well under normal lighting conditions, but the current version does not automatically turn on the phone's flash or automatically adjust the exposure of the camera to adapt to low light conditions. As a result, it may cause slight delays in marking attendance for the event.
7.2.4 No Offline Event Caching
The current Event Sphere application does not provide offline caching of event information. Once the app is opened, students are unable to browse event details and register for events without internet access. Implementing an offline-first approach with Firestore's offline persistence feature and offline device storage would greatly enhance the app's offline mode performance. 
7.2.5 Manual Fallback for QR Attendance
The existing system relies on Faculty to manually check and record attendance in the Faculty dashboard in the case of the QR code scanner or device camera not working on the event day. This creates a need for Faculty intervention for special cases of student attendance and may slow down the check-in process. The implementation of an alternative system such as an attendance PIN or manual attendance confirmation within the app would help to alleviate this.
7.3 Future Improvements
A number of improvements have been identified that will greatly enhance the functionality, user base and value of Event Sphere in future iterations:
7.3.1 iOS Version
To reach all of the students and faculty, Event Sphere should support iOS. Given that the system is developed using Flutter, multi-platform support for iOS devices can be achieved by resolving platform-specific issues, Firebase iOS SDK integration, Apple Push Notification Service (APNs) integration with FCM notifications and App Store installation procedures. Releasing the application for iOS will significantly boost its impact on campus.
7.3.2 AI-Based Event Recommendations
Event recommendations can be added to the Student interface using a machine learning (ML) model that makes recommendations based on student event registration history, department, interests and event attendance. This recommendation engine could leverage Firebase ML or Google Vertex AI and provide event recommendations on the Student Home Screen interface. This would enhance student engagement by recommending events that students are likely to be interested in.
7.3.3 Multi-Campus Support
Enhancing Event Sphere to support multiple university campuses would enable COMSATS University to roll out the system across all its campuses. Administrator namespaces would be available within the Firestore database for each campus, and administrators could choose to see events from other campuses. This feature would involve changes to the RBAC, Firestore database security rules and event filtering page to support segregation of data by campus, while providing a holistic administrative view at the institutional level.
7.3.4 Offline Mode with Event Caching
Enhancing the system with support for offline mode using Firestore's offline persistence settings and device caching would enable students to access event data (loaded previously), event registration details, and their QR attendance logs offline. This data would be automatically synchronized when the connection is restored. This would greatly enhance Event Sphere's resilience in the university setting with intermittent internet access.
7.3.5 Event Ticketing and Payment Integration
We can integrate a secure event ticketing and payment module to facilitate payment for university events, workshops and training. The payment module would support Faculty in marking events as paid and setting ticket prices, and would require payment confirmation before allowing students to register for events. This functionality would allow students to pay for events within the app through secure payment platforms like JazzCash, Easypaisa or Stripe. This would enable Event Sphere to be applied to events and workshops with a revenue stream for the university as well as outreach events for other institutions.
7.3.6 Web-Based Admin Dashboard
A custom-built web-based administrative dashboard for university administrators using the React.js framework and the Firebase Web SDK would offer a more feature-rich and screen-real-estate efficient way to manage users, view analytics, moderate content and create reports. A web dashboard would be an excellent complement to the mobile app, giving university administrators the ability to quickly and easily access desktop-class management features, especially for data-intensive operations like managing large groups of users, and reviewing analytics for multiple events.
7.3.7 Google Calendar Synchronization
Synchronization with Google Calendar API would enable students and faculty to add "approved" events to their Google Calendars with a single click from the Event Detail Screen. Events would display all pertinent information including event name, time, location and a link back to the Event Sphere event page. The new feature would enable users to plan their academic activities and minimize event no-shows by adding event reminders to their daily calendar activities.
7.3.8 Advanced Analytics and Institutional Reporting
Adding sophisticated data visualizations to the Reports and Analytics module, such as trend charts, heat maps, department engagement and event participation comparisons, and year-on-year event engagement analytics, would give university administrators much more useful institutional data. Exporting data to Google Data Studio or Power BI via Firebase would allow complex institutional reports to be generated, capable of being used in academic quality assurance processes and for university administration decision making. Email-based scheduled delivery of reports could also enhance the reporting process.
7.4 Final Remarks
Event Sphere is a valuable and feasible addition to the field of mobile academic event administration. The project successfully integrates:
Flutter and Dart mobile application development
Firebase real-time data cloud backend based development
Role-based authorization for the multiple stakeholders at a university
QR code scanning for quick and efficient check-in at events
Firebase Cloud Messaging for automatic real-time notifications
Event expense management process for budget management
Google Maps API to display event locations
Google Gemini for in-app user support
The system provides a real-world, user-friendly solution to a well-documented pain point of students, faculty and administration at COMSATS University Islamabad, Sahiwal Campus. Event Sphere's cloud-based automated and role-based system integrates manual administrative tasks into a digital platform, removing the existing inefficiencies that have prevented students, faculty and administrators from enjoying the benefits of student engagement, control and data-driven event planning that have become the standard in other industries.
Event Sphere is a foundational system that can be extended to support a future smart university event space - one that can evolve to include AI-based event recommendations, multi-campus institutional deployment, offline modes, integrated payments and analytics, thus providing a scalable and extensible platform for next-generation academic event management systems.
